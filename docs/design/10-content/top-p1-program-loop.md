# 欧米茄绝境战 P1：循环程序

本文档描述《最终幻想 XIV》绝本“欧米茄绝境战”P1 首个主要机制“循环程序”的模拟设计。英文资料名为 `The Omega Protocol (Ultimate)` / `Program Loop`。

当前目标是做机制练习内容，不做完整副本复刻。  
本文档中的机制规则应实现于 `packages/core` 与 `packages/content`，前端与服务端不得实现第二套战斗规则。
所有对玩家展示的技能、状态、提示与失败原因文案必须使用中文。英文名称仅作为资料原名、内部标识或代码注释使用，不进入玩家可见界面。

## 资料来源

- Icy Veins: The Omega Protocol Ultimate Phase 1 Guide
- Thaliak: The Omega Protocol (Ultimate)
- FFXIV Wiki: The Omega Protocol (Ultimate)
- Tuufless's FFXIV Notes: Omega / Program Loop

上述资料对核心机制描述一致：塔按 `1 -> 2 -> 3 -> 4` 处理，连线按 `3 -> 4 -> 1 -> 2` 处理；塔和连线都会附加破灭刻印，重复吃到会转为死亡宣告；连线额外附加体力衰减。

## 核心机制流程

### 总览

1. 首领读条循环程序。
2. 读条结束时，8 名玩家获得循环程序状态与第一至第四目标。
3. 每个编号随机分配给 2 名玩家。
4. 场上出现 2 座塔，同时首领给 2 名玩家生成连线。
5. 之后共进行 4 轮结算，每轮同时结算：
   - 当前轮的 2 座塔
   - 当前持有的 2 条连线
6. 塔处理顺序为第一目标、第二目标、第三目标、第四目标。
7. 连线处理顺序为第三目标、第四目标、第一目标、第二目标。
8. 第 4 轮结算后，若没有失败结果，机制完成。

### 时间轴

时间以战斗开始后毫秒计。

| 时间      | 事件                              | 数据                             |
| --------- | --------------------------------- | -------------------------------- |
| `0ms`     | 战斗开始                          | 首领位于中心                     |
| `6000ms`  | 首领开始读条循环程序              | 读条 `4000ms`                    |
| `10000ms` | 循环程序读条完成                  | 分配循环程序状态与第一至第四目标 |
| `10000ms` | 第 1 轮塔生成，初始连线生成       | 塔编号 1，连线目标随机           |
| `10000ms` | 首领开始读条冲击波                | 读条 `7600ms`                    |
| `17600ms` | 第 1 轮结算                       | 1 号踩塔，3 号吃线               |
| `17600ms` | 第 2 轮塔生成，首领再次读条冲击波 | 读条 `7600ms`                    |
| `26600ms` | 第 2 轮结算                       | 2 号踩塔，4 号吃线               |
| `26600ms` | 第 3 轮塔生成，首领再次读条冲击波 | 读条 `7600ms`                    |
| `35600ms` | 第 3 轮结算                       | 3 号踩塔，1 号吃线               |
| `35600ms` | 第 4 轮塔生成，首领再次读条冲击波 | 读条 `7600ms`                    |
| `44600ms` | 第 4 轮结算                       | 4 号踩塔，2 号吃线               |
| `47500ms` | 机制结束判定                      | 成功或失败                       |

说明：

- 资料页给出循环程序出现在 P1 `00:14`，并列出冲击波与塔判定在 `00:14`、`00:23`、`00:32`、`00:41` 发生。
- 为便于练习脚本从开场直接进入机制，本文档将读条开始设为 `6000ms`，让第一轮结算落在 `17600ms`。实现时可通过 `startOffsetMs` 调整。
- 后续轮次按 `9000ms` 间隔推进，贴合循环程序状态持续时间差值 `9s`。

## 状态与技能数据

### 循环程序状态

| 编号     | 玩家数 | 持续时间 | 塔处理轮次 |
| -------- | -----: | -------: | ---------: |
| 第一目标 |      2 |    `16s` |    第 1 轮 |
| 第二目标 |      2 |    `25s` |    第 2 轮 |
| 第三目标 |      2 |    `34s` |    第 3 轮 |
| 第四目标 |      2 |    `43s` |    第 4 轮 |

规则：

- 每个玩家获得一个循环程序编号。
- 持有循环程序状态的玩家可以踩塔。
- 塔只接受仍持有循环程序状态的玩家。
- 成功踩塔后移除该玩家的循环程序状态与目标编号。
- 循环程序状态自然到期且未被移除时，玩家获得遗忘。

### 遗忘

- 来源：循环程序状态到期未解除
- 持续时间：不单独模拟
- 效果：直接死亡
- 练习处理：获得该状态时立即写入失败原因并击杀玩家

说明：资料页写明遗忘会导致玩家无法执行动作并随机移动，实战上通常等同机制失败。第一版模拟不实现随机失控移动，直接按死亡处理。

### 冲击波

| 项       |                             数值 |
| -------- | -------------------------------: |
| 读条时间 |                         `7600ms` |
| 目标数   |                                2 |
| 伤害     |                          `35847` |
| 范围形状 | 以当前连线持有者为中心的圆形范围 |
| 范围半径 |                            `15m` |
| 附加状态 |               体力衰减、破灭刻印 |

规则：

- 两条连线在第 1 轮开始时生成。
- 连线不会在结算后消失，会保留到第 4 轮结束。
- 连线可以被其他玩家穿过并接走。
- 每轮冲击波结算时，以当前连线持有者为中心生成圆形伤害。
- 连线持有者与范围内其他玩家均受到伤害与状态。

说明：冲击波范围按首领目标环处理，半径为 `15m`。

### 体力衰减

| 项             |                     数值 |
| -------------- | -----------------------: |
| 持续时间       |                 `9000ms` |
| 最大生命值修正 | 第一版不模拟具体血量修正 |
| 生效时机       |         冲击波伤害结算后 |

规则：

- 先结算冲击波伤害，再附加体力衰减。
- 第一版不模拟最大生命值降低、当前生命值夹取或治疗补回。
- 持有体力衰减的玩家受到任何非 0 机制伤害时直接死亡。
- 体力衰减附加时不额外模拟血量下降。
- 3 号与 4 号吃线后，在其踩塔前体力衰减会结束；第一版只要求其不要在体力衰减期间再受伤。

### 破灭刻印

| 项           |                      数值 |
| ------------ | ------------------------: |
| 持续时间     |                 `10000ms` |
| 最大安全层数 |                         1 |
| 失败阈值     | 获得第 2 层时转为死亡宣告 |

规则：

- 冲击波与塔都会附加 1 层破灭刻印。
- 若玩家在状态未消失前再次获得破灭刻印，立即获得死亡宣告。
- 机制通过塔顺序与连线顺序错开，避免同一玩家连续承受两种判定。

### 死亡宣告

- 来源：破灭刻印达到 2 层
- 持续时间：模拟暂定 `3000ms`
- 效果：到期死亡
- 练习处理：获得死亡宣告时立即写入失败原因；是否延迟死亡由机制内核状态系统支持程度决定

### 塔判定

| 项       |                  数值 |
| -------- | --------------------: |
| 来源     |                塔结算 |
| 每轮塔数 |                     2 |
| 伤害     |               `35847` |
| 附加状态 |              破灭刻印 |
| 成功人数 | 每塔恰好 1 名有效玩家 |

规则：

- 每座塔必须由 1 名仍持有循环程序状态的玩家处理。
- 没有循环程序状态的玩家站在塔内不计入有效处理者。
- 有效处理者不足 1 人时，塔爆炸并团灭。
- 有效处理者超过 1 人时，判定为站位错误并团灭。
- 成功处理后移除处理者的循环程序状态与目标编号。

说明：资料描述塔必须由带循环程序状态的玩家处理，且不持有循环程序状态的玩家不计入塔处理。为提高练习反馈精度，第一版可把“多人有效踩同一塔”写成失败原因。

## 塔位生成

### 可选塔位

循环程序共有 8 个可能塔位。  
塔位分布在四个基本方向两侧，每个基本方向 2 个点。

塔位按“正方向锚点 + 切线偏移”生成：

- 正方向锚点半径：`15m`
- 每个正方向生成 2 座塔，位于该正方向两侧
- 两座塔的中心连线为场地圆的切线
- 每座塔中心相对正方向锚点沿切线偏移 `5m`
- 同一正方向的两座塔中心间距为 `10m`

坐标如下：

| 塔位 id | 正方向 | 切线偏移 | 塔中心坐标  |
| ------- | ------ | -------: | ----------- |
| `N_W`   | 北     |    `-5m` | `(-5, -15)` |
| `N_E`   | 北     |     `5m` | `(5, -15)`  |
| `E_N`   | 东     |    `-5m` | `(15, -5)`  |
| `E_S`   | 东     |     `5m` | `(15, 5)`   |
| `S_E`   | 南     |    `-5m` | `(5, 15)`   |
| `S_W`   | 南     |     `5m` | `(-5, 15)`  |
| `W_S`   | 西     |    `-5m` | `(-15, 5)`  |
| `W_N`   | 西     |     `5m` | `(-15, -5)` |

参数：

- 正方向锚点半径：`15m`
- 切线偏移距离：`5m`
- 塔判定半径：`5m`
- 塔可视半径：`5m`
- 塔预显时间：上一轮塔判定的同时生成下一轮塔，不做半透明预显

说明：

- 资料可确认共有 8 个塔位，四个方向各有两个偏移点。
- 资料可确认每轮 2 座塔不会出现在同一个基本方向，并且两塔所属正方向夹角为 `90deg` 或 `180deg`。
- 该坐标定义使塔靠近外圈，塔外缘可贴近死亡墙，符合“正方向两侧紧贴生成”的视觉布局。

### 生成规则

每轮生成 2 座塔：

- 从 8 个可选塔位中选择。
- 本机制内同一个塔位不重复使用。
- 同一轮两座塔不能属于同一个基本方向。
- 同一轮两座塔所属正方向夹角必须为 `90deg` 或 `180deg`。
- 4 轮后 8 个塔位全部用完。

基础实现算法：

1. 洗牌 8 个塔位。
2. 逐轮选择 2 个未使用塔位。
3. 校验两塔不在同一基本方向。
4. 校验两塔所属正方向夹角为 `90deg` 或 `180deg`。
5. 若无法满足，重新生成本机制塔序列。

为降低练习随机难度，第一版也可以使用固定塔序列，并在后续加入随机：

| 轮次 | 塔位         |
| ---: | ------------ |
|    1 | `N_W`、`S_E` |
|    2 | `E_N`、`W_S` |
|    3 | `N_E`、`S_W` |
|    4 | `E_S`、`W_N` |

## 连线传递

### 初始连线

- 第 1 轮开始时，首领生成 2 条连线。
- 当前实现会随机分配循环程序编号。
- 初始目标为本局第二目标的两名玩家。
- 初始目标不是最终吃线责任者；玩家需要通过移动把线交给对应编号。

### 传递规则

第一版推荐实现为“接触式传递”：

- 连线以首领与当前持有者之间的线段表示。
- 允许目标中的其他存活玩家与该线段距离 `<= 0.6m`，且该玩家距离当前持有者 `> 0.6m` 时，可接走连线。
- 同一条连线传递后进入 `500ms` 冷却，避免同帧抖动。
- 每名玩家同一时刻最多持有 1 条连线。
- 若两个玩家同帧满足接线条件，选择距离线段最近者；距离相同则按小队顺序。
- 当前 TOP 跑法按本局随机结果把两条线拆为两个允许目标序列：`第二目标 A -> 第三目标 A -> 第四目标 A -> 第一目标 A -> 第二目标 A`、`第二目标 B -> 第三目标 B -> 第四目标 B -> 第一目标 B -> 第二目标 B`。这是为了让第一版机器人不需要复杂绕线智能；核心连线机制支持不配置允许目标的开放传递。

### 连线结算顺序

| 轮次 | 塔处理编号 | 连线处理编号 |
| ---: | ---------- | ------------ |
|    1 | 第一目标   | 第三目标     |
|    2 | 第二目标   | 第四目标     |
|    3 | 第三目标   | 第一目标     |
|    4 | 第四目标   | 第二目标     |

失败条件：

- 当前轮结算时，任意连线持有者不是本轮指定连线编号。
- 两条连线由同一个玩家持有。
- 冲击波命中正在踩塔的玩家。
- 冲击波互相重叠时不单独写特殊死亡规则，死亡判定统一按破灭刻印叠层处理：破灭刻印达到 2 层或以上时转为死亡宣告。

## 推荐解法

### 基本原则

- 当前轮踩塔的 2 人去塔。
- 当前轮吃线的 2 人把线带到远离塔的位置。
- 其他人靠近塔位附近但避开冲击波范围。
- 吃线编号比踩塔编号晚两步：`3 -> 4 -> 1 -> 2`。

### 轻队优先级解法

使用两组固定优先级决定“谁去哪个方向的塔 / 谁把线带到哪个方向”。

推荐优先级：

- 一组：从西北开始逆时针
- 二组：从北开始顺时针
- 调整优先级：坦克 -> 近战 -> 远程 -> 治疗

项目标准槽位可写成：

- 一组：`MT`、`H1`、`D1`、`D3`
- 二组：`ST`、`H2`、`D2`、`D4`

若同一编号的两人落在同一轻队，需要按调整优先级让其中一人去另一侧，确保每轮两个责任者分别处理两座塔或两条线。

### 机器人跑法

第一版机器人只做确定性脚本，不实现智能决策。  
机器人从自身循环程序状态和当前连线允许目标序列判断本局职责，可以写死坐标、写死移动时间点，并假设其他玩家与机器人均按既定方案正确执行。测试用例使用固定随机序列保证可复现。

1. 战斗开始时读取自身循环程序编号。
2. 根据本轮职责判断：
   - 当前编号等于塔编号：移动到分配塔中心。
   - 当前编号等于连线编号：穿过对应车道的当前连线，接线后移动到空方向外侧。
   - 否则：移动到安全待机点。
3. 3/4 号吃线后，在体力衰减结束前避免进入任何伤害范围。
4. 3/4 号准备踩塔前，只需要确认其体力衰减已经结束；第一版不模拟治疗补血。

建议待机点：

- 无职责玩家在场中安全点待机，避开当前塔位与冲击波范围。
- 连线玩家固定在半径 `17m` 的位置吃判定，方向选择远离当前塔组，确保冲击波半径 `15m` 不覆盖塔组。
- 踩塔玩家站在塔中心。

接线移动原则：

- 上一轮接线人在结算后原地等候下一轮接线人前来接线。
- 下一轮接线人尽量靠场中移动，优先保持在半径 `10m` 内，降低误接线概率。
- 无职责玩家尽量靠场外移动，优先保持在半径 `10m` 外，避开接线路径。
- 接线人在判定前移动到半径 `17m` 的固定外圈点，确保冲击波远离塔组。
- 第 1 轮点名出现后，除第 1 轮接线人外，所有人后退，确保第 1 轮接线人能第一时间接到线。
- 第一版机器人控制可以先写死时间点，在确定的时间点移动、等待、接线和出场。

## 失败条件

机制失败时调用 `state.fail(reason)`，不中断后续脚本，最终统一结算。

失败原因建议：

- 循环程序状态到期未解除：`{actor} 未及时踩塔解除循环程序`
- 错号踩塔：`{actor} 在错误轮次踩塔`
- 塔无人处理：`第 {round} 轮塔未被处理`
- 多人踩同塔：`第 {round} 轮塔被多人处理`
- 连线错误编号：`第 {round} 轮连线由错误编号处理`
- 冲击波命中塔组：`第 {round} 轮冲击波命中踩塔玩家`
- 破灭刻印叠层：`{actor} 重复承受循环程序判定并获得死亡宣告`
- 体力衰减期间受伤：`{actor} 在体力衰减期间受到伤害`
- 越界死亡：`{actor} 触碰死亡墙`

## 需要扩展的机制实现

### 机制内核扩展

扩展原则：能交给内容层自定义的规则不写死在机制内核中，机制内核提供通用回调与原子结算能力。

- 状态系统提供受伤前后回调，由内容层判断体力衰减、破灭刻印叠层、死亡宣告等具体规则。
- 状态系统支持叠层阈值触发内容层回调。
- 塔机制提供处理回调，由内容层判断哪些玩家是有效处理者。
- 塔机制支持按有效处理者数量结算。
- 塔机制支持未处理时触发内容层失败回调。
- 连线机制支持首领到玩家的持久线段。
- 连线机制支持接触式传递与传递冷却。
- 范围结算支持同一时刻多个来源分别命中；每个来源保持原子结算，并按确定顺序逐个处理。
- 脚本状态支持记录循环程序编号、塔序列、连线持有者。

### 内容层扩展

需要新增以下内容定义：

- `top_p1_program_loop` 战斗定义。
- 循环程序时间轴。
- 循环程序编号随机分配器。
- 塔位序列生成器。
- 连线目标与传递规则配置。
- 机器人跑法与优先级解析。
- 失败原因文本。

文件组织要求：

- 新机制必须单独写文件，不继续堆在 `packages/content/src/index.ts`。
- 现有演示机制也需要拆出独立文件。
- `packages/content/src/index.ts` 只保留类型导出、战斗注册表与聚合导出。
- 推荐结构：
  - `packages/content/src/battles/opening-two-rounds.ts`
  - `packages/content/src/battles/top-p1-program-loop.ts`
  - `packages/content/src/battles/index.ts`
  - `packages/content/src/bot/top-p1-program-loop.ts`

### 测试要求

实现完成后必须补充测试用例验证机器人跑法。

最低测试要求：

- 8 个槽位全部由机器人控制时，固定随机种子或固定机制脚本能够完成循环程序。
- 机器人能按固定时间点完成接线、踩塔、出场吃冲击波。
- 冲击波不会命中当前塔组。
- 塔判定只统计持有循环程序状态的有效处理者。
- 体力衰减期间机器人不会再次受到机制伤害。
- 若某个关键路径不知道如何稳定验证，先停止并询问，不用猜测实现。

## 已确认实现取舍

- 场地半径第一版固定为 `20m`。
- 塔允许多人进入，但只按持有循环程序状态的玩家计算有效处理者。
- 第一版实现可传递连线。
- 机器人控制使用固定坐标与固定时间点移动，不需要智能决策。
- 第一版暂不复刻治疗输入与真实生命值上限修正；体力衰减期间受伤即死，不模拟补血。
- 冲击波展示名使用“冲击波”。
- 塔机制展示名使用“塔判定”。
