# 房间生命周期

## 房间阶段

房间生命周期如下：

1. `waiting`
2. `running`
3. `closed`

## waiting

该阶段表示房间处于待开始状态。  
客户端在进入房间后立即显示模拟页，但机制时间轴不推进。

该阶段允许：

- 选择战斗
- 切换槽位
- 玩家准备
- 房主开始
- 玩家移动与转向同步

该阶段规则如下：

- 房主固定视为已准备
- 其他玩家切槽、切战斗、战斗结束后重置为未准备
- 切换战斗后立即重建等待态场景
- 槽位始终补满为 8 人，不足位置由 Bot 占位

## running

该阶段表示房间正在推进权威模拟。

该阶段允许：

- 玩家输入移动、转向、防击退

该阶段禁止：

- 切换战斗
- 切换槽位
- 新玩家加入

当战斗脚本完成后：

- 房间立即回到 `waiting`
- 保留上一轮结果
- 不跳转独立结算页
- 直到下一次开始或切换战斗时才整体重置

## closed

该阶段表示房间被销毁。

触发条件如下：

- 房主主动离开
- 房主断开 WebSocket 连接

关闭后效果如下：

- 其他玩家被移出房间
- 前端返回大厅页

## 离房与补位规则

- 非房主在 `waiting` 阶段离开或断线：立即由 Bot 补位
- 非房主在 `running` 阶段离开或断线：本轮先保留原角色，回到 `waiting` 时再补 Bot
- 当前已死亡的角色在任何阶段都不能移动

## 模拟实例关系

- `waiting` 与 `running` 都以 `packages/core` 的权威模拟实例作为唯一状态源
- `waiting` 阶段不再维护独立等待态移动实现，连续移动仍走同一套输入入队、Tick 消费与事件同步链
- 每次切回待开始或重新开始时，服务端会按当前房间成员和保留策略重建等待态或运行态模拟状态
