# 房间生命周期

## 房间阶段

房间生命周期如下：

0. `pending`
1. `waiting`
2. `running`
3. `closed`

## pending

该阶段表示房主已经提交建房申请，但真实房间尚未实例化。

该阶段规则如下：

- `POST /rooms` 只创建建房申请并返回 `roomId` 和 `expiresAt`
- 建房申请不出现在公开房间列表中
- 只有房主本人可以通过 `room:join` 消费建房申请
- 房主成功加入后，服务端实例化真实房间并进入 `waiting`
- 建房申请 `30s` 内未被房主消费则过期

## waiting

该阶段表示房间处于待开始状态。  
客户端在进入房间后立即显示模拟页，但机制时间轴不推进。

该阶段允许：

- 选择战斗
- 切换槽位
- 切换观战
- 房主开始倒计时
- 玩家移动与转向同步

该阶段规则如下：

- 切换战斗后立即重建等待态场景
- 槽位始终补满为 8 人，不足位置由 Bot 占位
- 玩家可以从大厅直接加入观战，直接观战不占用战斗槽位，槽位满员时仍可加入
- 玩家切换观战后不占用槽位，原槽位立即由 Bot 占位
- 观战玩家仍属于房间成员，继续接收同步，点击可用槽位后回到场内
- 房主观战后仍可开始倒计时；开始条件只要求当前连接属于房主、房间处于 `waiting` 且已选择战斗
- 开始倒计时默认为 `5s`，房主可配置为 `1s` 到 `30s`
- 开始倒计时时服务端先重置为标准开场状态
- 倒计时数字由服务端通过 `room:countdown` 每秒广播
- 倒计时期间仍保持 `waiting`，但不允许切换战斗、切换槽位、切换观战或新玩家加入战斗槽位
- 倒计时期间的玩家移动与朝向会保留到正式开战
- 倒计时结束后服务端正式切入 `running`，战斗时间从 `0ms` 开始
- 允许 8 个槽位均为 Bot

## running

该阶段表示房间正在推进权威模拟。

该阶段允许：

- 玩家输入移动、转向、防击退

该阶段禁止：

- 切换战斗
- 切换槽位
- 新玩家加入

当战斗脚本完成后：

- 房间立即回到 `waiting`
- 保留上一轮结果
- 不跳转独立结算页
- 直到下一次开始或切换战斗时才整体重置

## closed

该阶段表示房间被销毁。

触发条件如下：

- 房主主动离开
- 房主断开 WebSocket 连接

关闭后效果如下：

- 其他玩家被移出房间
- 前端返回大厅页

## 离房与补位规则

- 非房主在 `waiting` 阶段离开或断线：立即由 Bot 补位
- 非房主在 `running` 阶段离开或断线：本轮先保留原角色，回到 `waiting` 时再补 Bot
- 当前已死亡的角色在任何阶段都不能移动

## 模拟实例关系

- `waiting` 与 `running` 都以 `packages/core` 的权威模拟实例作为唯一状态源
- `waiting` 与 `running` 的移动链路永远不分叉
- 两个阶段都使用同一套客户端本地移动、`sim:input-frame` 位姿样本上报、服务端去重与 Tick 写入链路
- 每次切回待开始或重新开始时，服务端会按当前房间成员和保留策略重建等待态或运行态模拟状态
