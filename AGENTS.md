# FF14 Arena Next 项目规约

## 1. 项目定位

本项目是一个多人联机机制模拟项目。

项目的详细设计、技术说明和相关文档统一从 [docs/README.md](./docs/README.md) 进入。  
实现时以仓库中的现行文档为准，不自行发明与文档冲突的新规则。

## 2. 工作语言

- 面向用户的说明、提问、建议与总结使用中文。
- 代码注释和项目文档默认使用中文，除非代码上下文明确要求英文。
- 文档和实现说明使用直接、确定的口吻，不使用摇摆或模糊表述。

## 3. 开工规则

- 开始实现前，先阅读 [docs/README.md](./docs/README.md) 和与当前任务直接相关的文档。
- 若发现代码现状与文档存在冲突，先停止实现，向用户说明冲突点。
- 不在未同步文档的情况下擅自偏离现有设计。

## 4. 架构红线

- 战斗规则只写在 `packages/core`。
- `apps/web` 负责页面、交互、展示与客户端镜像，不承载战斗规则。
- `apps/server` 负责房间、同步和权威模拟托管，不实现第二套战斗规则。
- 内容层只存放战斗内容、脚本和配置，不承载引擎逻辑。
- 浏览器端与服务端共享同一套 `core`，不维护两套独立规则实现。
- 项目统一使用 `pnpm` 作为包管理器，统一使用 `Node.js` 作为运行时。

## 5. 变更同步规则

- 修改全局规则时，同步更新相关文档。
- 实现功能或修改功能后，按需补充相关文档，确保实现与文档说明保持一致。
- 修改术语时，全局统一，不只改一处。
- 修改某个模块时，检查它影响到的相关模块与文档，避免局部改动破坏整体一致性。

## 6. 禁止项

- 禁止把临时规则硬编码进 Vue 组件、Pixi 场景对象、Socket 回调或路由处理器。
- 禁止绕过既定架构边界，把规则散落到前端或服务端壳层。
- 禁止在当前阶段擅自扩展完整职业系统、复杂数值系统或复杂 Bot 系统，除非用户明确要求。
- 禁止用“先这样凑合”的方式引入与现有文档冲突的实现。
- 禁止在仓库中混用多种包管理器配置。

## 7. Git 提交格式

- Git Commit Message 使用 Conventional Commits 格式。
- 提交格式为：`type: subject`
- `type` 使用英文小写，例如：`feat`、`fix`、`docs`、`refactor`、`chore`、`test`
- `subject` 使用中文，直接描述本次改动
- 单次提交只包含一个清晰的逻辑主题，不混入无关改动

示例：

- `feat: 初始化 monorepo 目录结构`
- `fix: 修正战斗结果聚合逻辑`
- `docs: 补充前端操控模式说明`

## 8. 文档入口

开发前先看 [docs/README.md](./docs/README.md)。

文档编写与维护遵循以下要求：

- 文档不应无节制增长。
- 若单篇文档过长且内容可拆分，则拆分为子文档，并在入口文档中保持清晰的层级与导航。
- 文档结构应清晰，便于按主题定位、阅读和后续维护。
